# MVP 立项说明

## 项目定位

本项目定位为“养老院安全康复系统第一代 MVP 试点版本”，目标是在一个养老院或一个楼层内跑通最小闭环：

```text
老人档案 -> 护理任务 -> 设备/人工告警 -> 护理响应 -> 记录留痕 -> 管理看板
```

MVP 阶段不追求一次性做成完整智慧养老平台。先验证业务流程、数据模型、接口边界和现场可用性，再逐步接入真实设备、AI 视频识别、家属端和多院区管理。

## MVP 必须包含

- 账号登录与角色基础权限。
- 老人基础档案。
- 护理任务列表与状态展示。
- 告警记录列表与基础分级。
- 摄像头/设备台账展示。
- 管理端运营看板。
- 数据库存储与初始化种子数据。
- 健康检查接口。
- 统一接口返回、错误和日志规则。

## MVP 暂不包含

- 真实 AI 跌倒识别模型。
- 真实 RTSP 视频转码播放。
- 真实 MQTT/ONVIF/GB28181 设备接入。
- 家属端独立应用。
- 医保、HIS、监管平台深度对接。
- 多院区、多租户、复杂计费。
- 生产级高可用集群。

这些能力必须先在架构文档中定义边界，再逐步做骨架和实现。

## 试点验收目标

- 系统可以在本机或试点服务器稳定启动。
- 管理员可以登录后台。
- 看板数据来自数据库，而不是纯前端写死。
- 老人、任务、告警、摄像头数据模型可以支撑后续扩展。
- 后端代码按模块分层，不继续把所有业务堆在一个文件里。
- 后续接设备和 AI 时，不需要推倒当前后端。

## 试点产品化范围

当前从“演示系统”升级到“可试点产品”，只补最小业务闭环：

```text
告警闭环：新告警 -> 确认 -> 处理中 -> 已解决 / 误报
护理任务：待处理 -> 进行中 -> 已完成 / 超时 / 已复核
设备台账：设备在线状态、心跳时间、绑定老人/区域
AI事件：接收事件、置信度、人工确认、误报标记
```

本阶段仍不实现真实视频推理、MQTT 网关、家属端 App、医保/HIS 对接。所有设备与 AI 能力只做标准接口和最小记录能力，用于试点流程验证。

## 技术路线锁定

MVP 阶段锁定以下技术：

```text
前端：HTML + CSS + 原生 JavaScript
后端：NestJS + TypeORM
数据库：MariaDB/MySQL，本机开发可兼容 PostgreSQL
认证：JWT
运行环境：Node.js
```

不引入 Vue、React、Spring Boot、微服务、Kubernetes 等重型技术，除非出现明确业务原因并更新架构文档。

## 变更原则

任何技术、目录、依赖、数据库、接口规范变更，都必须说明：

- 为什么要变。
- 影响哪些模块。
- 是否破坏现有接口。
- 是否需要迁移数据。
- 是否更新架构文档。
