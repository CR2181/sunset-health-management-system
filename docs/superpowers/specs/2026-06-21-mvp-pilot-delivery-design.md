# 养老院安全康复系统 MVP 试点交付设计

## 1. 目标

在现有原生前端、NestJS、TypeORM 和 MySQL/PostgreSQL 技术栈上，形成可用于单院区 30 张床试点的第一代 MVP。系统必须能够通过数据库完成老人档案、护理任务、告警和设备台账的最小闭环，并提供后端权限控制、审计留痕、部署资料和演示流程。

本次不更换前端框架，不引入微服务、消息队列、Kubernetes 或复杂权限平台。

## 2. 已确认的试点边界

### 2.1 本次交付

- 登录、退出、会话恢复和后端 JWT 认证。
- 超级管理员、院长、护理员、康复师、家属、访客六类角色。
- 工作台、老人档案、护理任务、安全告警、告警记录、设备台账和审计日志。
- 护理任务从待处理到进行中、完成的状态闭环。
- 告警从新建到确认、解决或误报的状态闭环。
- 设备在线状态、最近心跳、电量和绑定老人信息。
- 30 张床的虚构、脱敏、可重复初始化的试点数据。
- 部署、备份、验收、培训、隐私和演示资料。

### 2.2 本次不交付

- 真实 AI 模型推理及医疗诊断。
- 真实 RTSP 转码和视频播放。
- MQTT、ONVIF、GB28181 正式设备网关。
- 家属独立 App、微信小程序或短信平台。
- HIS、医保、长护险正式生产对接。
- 多院区、多租户、自动计费和生产级高可用集群。

预留的 AI、设备和第三方接入接口只能用于模拟事件和边界验证，不宣传为正式设备能力。

## 3. 技术方案

### 3.1 技术栈保持不变

```text
前端：HTML + CSS + 原生 JavaScript
后端：NestJS + TypeORM
数据库：MySQL/MariaDB 或 PostgreSQL
认证：JWT
运行环境：Node.js
```

选择原因：现有代码已经采用这些技术，能够覆盖 30 床试点的数据量和访问量。当前优先事项是闭合业务流程和安全边界，不是扩大技术复杂度。

### 3.2 数据流

```text
浏览器页面
  -> 携带 JWT 请求 NestJS API
  -> JWT 守卫验证身份
  -> 角色和数据范围守卫验证权限
  -> Service 执行业务规则
  -> TypeORM 读取或写入数据库
  -> 写操作记录审计日志
  -> 统一响应返回前端
```

老人、任务、告警、设备、摄像头、审计和工作台统计不得从前端静态数组作为正式数据源。后端不可用时，页面显示明确的服务不可用状态，不静默回退为本地业务数据。

演示账号说明可以保留在开发配置中，但生产部署必须关闭账号密码展示。

## 4. 前端信息架构

统一路由注册表继续作为菜单、标题、面包屑和权限的单一来源。每个主菜单保持唯一 `key` 和唯一 `path`，当前路由先精确匹配，再使用最长安全前缀匹配，最终只产生一个 `activeMenuKey`。

试点核心页面：

```text
/dashboard       角色工作台
/residents       老人档案
/care-tasks      护理任务
/safety-alerts   当前待处理告警
/alert-records   历史告警记录
/devices         摄像头与设备台账
/audit-logs      审计日志（管理员和院长）
```

康复、家属、报表、设置、第三方接入和访客演示保留独立路由。未进入本次业务闭环的页面使用独立建设中状态，不得跳转到其他模块或伪装成已完成页面。

## 5. 后端角色和权限

前后端统一角色代码：

```text
super_admin  超级管理员
director     院长
nurse        护士/护理员
rehab        康复师
family       家属
visitor      第三方/访客
```

后端是最终授权主体。所有包含老人、护理、告警、设备和审计数据的接口必须先认证，再进行角色判断。

最小权限规则：

| 角色 | 允许范围 |
| --- | --- |
| 超级管理员 | 全部试点数据、配置和审计日志 |
| 院长 | 全院只读概览、告警、任务、设备和审计日志；不管理超级管理员 |
| 护理员 | 自己负责的老人、护理任务和安全告警处理 |
| 康复师 | 授权老人的康复相关摘要；不处理护理任务和系统设置 |
| 家属 | 只读绑定老人的脱敏摘要 |
| 访客 | 只读虚构演示数据，不访问真实业务接口 |

第一代 MVP 在用户表中保存可选的 `boundResidentCode` 和责任范围。家属必须按绑定老人过滤；护理员按责任范围过滤。不能只依赖前端隐藏菜单实现数据隔离。

## 6. 审计设计

审计日志保存以下字段：

- 操作时间。
- 操作者 ID、账号和角色。
- 动作代码。
- 资源类型和资源 ID。
- 操作摘要。
- 不含密码、令牌、完整手机号的安全元数据。

必须记录：

- 登录成功和登录失败。
- 老人档案创建和修改。
- 护理任务状态变更。
- 告警确认、解决和误报。
- 设备心跳、设备事件和 AI 模拟事件。
- 审计日志查询。

权限拒绝在应用结构化日志中记录请求 ID、路径和角色，不记录 JWT 原文。试点阶段不为普通只读列表的每次自动刷新写数据库审计，避免日志噪声；敏感审计日志查询例外。

## 7. 30 床试点数据

试点数据全部为虚构数据，覆盖：

- 30 位老人，分布在 3 个护理区域和 30 张床位。
- 不同护理等级、风险标签和健康摘要。
- 6 类试点账号及家属绑定关系。
- 每位老人至少 1 条护理任务，包含待处理、进行中、完成和超时状态。
- 高、中、低三级告警及新建、确认、解决、误报状态。
- 摄像头、呼叫器、智能床垫、烟感和可穿戴设备台账。
- 可用于统计工作台的稳定数据分布。

种子数据采用确定性业务编号，并在已有数据时安全跳过或按业务编号补齐。任何真实姓名、手机号、身份证号、病历号或真实视频地址不得进入仓库。

## 8. 错误处理和可用性

- 参数错误返回 400。
- 未登录返回 401。
- 无权限返回 403。
- 资源不存在返回 404。
- 冲突返回 409。
- 未预期错误返回 500，响应中不暴露堆栈和数据库细节。
- 前端对 401 清理失效会话并返回登录页。
- 前端对 403 显示无权限页。
- 数据库或 API 不可用时显示可恢复的错误状态和重试入口。

所有 API 保持现有统一响应包装和全局异常过滤器。

## 9. 测试与验收

交付前至少验证：

- 路由注册表不存在重复路径和重复菜单键。
- 每个角色看到的菜单、默认工作台和手动 URL 权限正确。
- 未登录不能读取敏感 API。
- 家属只能读取绑定老人信息。
- 访客不能读取真实老人、告警、任务、设备和审计数据。
- 核心页面数据由数据库 API 提供，API 失败时不出现静态业务数据。
- 任务、告警和设备操作写入数据库并生成审计日志。
- 30 位老人及关联数据可以重复初始化而不重复。
- 后端构建、依赖审计和试点冒烟测试通过。
- Windows 本机部署、启动、停止、备份和恢复步骤可执行。

## 11. 完成标准

只有同时满足以下条件才视为本轮完成：

1. 已确认的 MVP 范围写入文档并与代码一致。
2. 核心数据来自数据库，前端不再静默伪造正常状态。
3. 前后端角色代码统一，敏感接口均受后端保护。
4. 核心写操作和登录行为可在审计日志中追溯。
5. 30 床脱敏数据、部署资料、验收表、培训资料和演示脚本齐全。
6. 自动检查、后端构建、安全审计和人工浏览器验证均有通过证据。
