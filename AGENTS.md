# 项目长期执行规则

本文件适用于整个“养老院安全康复系统”仓库。所有开发、修复、文档更新和自动化任务都必须遵守以下规则。

## 完成代码修改后的 GitHub 自动提交与推送

每次完成代码修改并通过相应测试或启动检查后，必须自动将本次任务的改动提交到本地 Git，并推送到 GitHub 当前分支。不得只修改文件而不说明本地提交和远程推送状态。

### 1. 提交前检查

必须先执行：

```powershell
git status
git diff --stat
git diff --name-status
```

逐项确认改动属于当前任务，并检查是否包含敏感信息、运行产物或无关用户改动。

### 2. 暂存与提交

当工作区中的全部改动都属于当前任务，且确认不包含禁止提交内容时，执行：

```powershell
git add .
git commit -m "简短说明本次改动"
```

如果工作区存在其他任务、用户本人或其他智能体留下的未提交改动，禁止使用 `git add .`。此时必须只暂存当前任务明确修改的文件，避免误提交、覆盖或泄露无关内容。

提交信息必须简短、准确，说明本次实际改动，不得使用含糊的 `update`、`changes` 等描述。

### 3. 自动推送当前分支

提交成功后执行：

```powershell
git push origin HEAD
```

### 4. 首次推送分支

如果当前分支尚未关联远程分支，执行：

```powershell
git push -u origin HEAD
```

禁止自动强制推送。不得执行 `git push --force` 或 `git push --force-with-lease`，除非用户明确授权。

### 5. 推送失败处理

如果 push 失败，必须明确告诉用户真实原因，不得假装上传成功。常见原因包括：

- GitHub 没有登录
- 当前账号没有 push 权限
- 网络连接失败
- 分支受到保护
- 远程分支包含本地没有的提交，需要先安全检查和同步
- 远程仓库地址或认证配置错误

遇到需要 pull、merge 或 rebase 的情况，必须先检查远程差异和本地未提交改动，不得直接覆盖用户代码。

### 6. 禁止提交的内容

不得提交：

- `node_modules`
- `dist`
- `.env`
- `*.log`
- `backend/node_modules`
- `backend/dist`
- `backend/.env`
- `backend/.runtime`
- 数据库运行文件、数据目录、备份文件和临时文件
- 真实老人、家属、员工或机构隐私数据
- API 密钥、访问令牌、私钥和其他机密信息
- 真实生产账号和密码

提交前必须确认 `.gitignore` 能覆盖运行产物，并使用 `git diff --cached --name-status` 复核最终暂存清单。

## 结果报告

每次自动提交和推送后，必须向用户报告：

- 当前分支名称
- 本地提交 SHA 和提交信息
- push 是否成功
- 远程分支名称
- 未提交或未推送的剩余改动
