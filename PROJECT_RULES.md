# 养老院安全康复系统统一项目规则

本文件把当前项目中的 README、AGENTS.md 和 `docs/` 规则集中到一个地方，方便开发、试点、验收和后续维护。

本文件不会删除或覆盖已有规则。遇到本文件末尾列出的冲突或“待确认”事项时，先暂停相关改动并请项目负责人确认。

## 1. 项目定位

- 本项目当前定位为“养老院安全康复系统第一代 MVP 试点版本”。
- 当前主要服务对象是养老院管理人员、院长或运营负责人、护士长、护理员、医生或康复师以及系统管理员。
- 家属目前只考虑授权范围内的有限信息查看，不把独立家属 App 作为第一阶段交付内容。
- 第一阶段不是建设完整智慧养老平台，而是在一家养老院、一个楼层或一个护理单元内跑通最小业务闭环。
- 第一阶段核心闭环是：

```text
老人档案 -> 护理任务 -> 人工/设备/AI事件 -> 告警响应 -> 人工复核 -> 审计留痕 -> 管理看板
```

- 第一阶段先验证流程是否可用、数据是否能保存、权限是否有效、问题是否能追溯，再决定是否扩大范围。
- 项目的正式中文名称目前存在多种写法，统一名称见《待确认问题》；确认前不要擅自修改品牌名称和仓库名称。

## 2. MVP 边界

### 必须执行

- 实现账号登录和基础角色权限。
- 老人档案必须来自数据库，并有稳定业务编号。
- 护理任务必须能查看和更新状态，并留下操作记录。
- 告警必须支持查看、确认、解决和标记误报。
- 设备和摄像头先建立台账，至少能记录在线状态、心跳时间、位置和绑定关系。
- AI 事件先做事件接收、记录、人工复核和误报标记，不做真实模型承诺。
- 关键写操作必须写入审计日志。
- 管理看板必须读取后端和数据库汇总数据，不能用前端固定数字冒充实时数据。
- 必须提供健康检查、统一返回格式、错误处理和基础日志能力。
- 家属、护理员等受限角色只能访问授权范围内的数据，最终权限必须由后端校验。
- 每次修改都要保证现有可运行能力不被破坏。

### 建议执行

- 使用 20–50 张床的完全虚构试点数据验证流程。
- 护理员网页优先适配手机和平板，管理端优先适配电脑浏览器。
- 为告警、任务、设备和 AI 事件增加清晰的状态筛选和空数据提示。
- 在试点前完成数据库备份和恢复演练。
- 用周报记录任务完成率、告警响应、误报、设备离线和使用反馈。
- 所有未完成模块显示明确的“建设中”或“暂不可用”，不要跳到无关页面。

### 暂不做

- 不做真实 AI 模型训练和生产级推理。
- 不做真实 RTSP 视频流的生产级接入、转码和多路视频墙。
- 不做真实 MQTT、ONVIF、GB28181 或厂商设备协议接入。
- 不开发原生 Android、iOS、Flutter、React Native 家属端或护理端 App。
- 不做 HIS、医保、长护险、民政监管平台深度对接。
- 不做多院区、多租户、复杂计费和生产级高可用集群。
- 不引入微服务、Kubernetes 或复杂消息队列。
- 不把长期建设方案中的完整财务、药品、膳食、门禁、定位和运营能力当成第一代 MVP 已完成内容。

## 3. 技术栈规则

### 前端

- 使用 HTML、CSS 和原生 JavaScript。
- 当前不引入 React、Vue、Vite、Webpack 等新框架或构建系统。
- 前端采用轻量页面模块、统一路由、统一权限和统一 API 请求方式。

### 后端

- 使用 Node.js、NestJS 和 JWT 认证。
- 优先使用 NestJS 自带的模块、守卫、管道、异常和依赖注入能力。

### 数据库与 ORM

- 当前优先使用 MariaDB / MySQL。
- ORM 使用 TypeORM。
- PostgreSQL 兼容能力可以保留，但当前试点默认数据库仍是 MariaDB / MySQL。

### 启动方式

- 完整本机服务优先使用：

```powershell
cd backend
npm run service:start
```

- 停止完整服务使用：

```powershell
cd backend
npm run service:stop
```

- 后端默认地址为 `http://127.0.0.1:3000`，API 前缀为 `/api`。

### 环境变量

- 端口、数据库类型、数据库地址、账号、密码、数据库名和 JWT 密钥必须通过环境变量或 `.env` 管理。
- 至少包括：`PORT`、`DB_TYPE`、`DB_HOST`、`DB_PORT`、`DB_USERNAME`、`DB_PASSWORD`、`DB_DATABASE`、`JWT_SECRET`、`DB_SYNC`。
- `.env` 不得提交，仓库必须提供不含真实秘密的 `.env.example`。

### 技术变更

- 未经项目负责人明确批准，不得引入 React、Vue、Flutter、React Native、Android、iOS、Spring Boot、微服务、Kubernetes 等重型技术。
- 技术、依赖、目录、接口或数据库方案发生变化前，必须说明原因、影响范围、迁移方法和回退方式，并更新架构文档。

## 4. 架构规范

- 接口层由 controller 负责，只处理 HTTP 请求、响应、身份信息和参数接收。
- 业务层由 service 负责，处理任务状态、告警处置、权限范围和业务规则。
- 数据层由 entity、TypeORM Repository 和必要的查询封装负责。
- DTO 负责请求参数结构和 `class-validator` 校验。
- module 只组织当前业务模块的依赖，不承载业务逻辑。
- common 只放通用能力，不放具体老人、告警或设备业务。
- 配置统一从环境变量读取，不在源码中写数据库密码、JWT 密钥或设备密码。
- 所有 API 使用 `/api` 前缀和 JSON 数据。
- 成功响应逐步统一为 `success`、`code`、`message`、`data`。
- 错误响应使用正确 HTTP 状态码，不允许所有错误都返回 200。
- 异常优先使用 NestJS 的标准异常类，不随意抛普通字符串。
- 后端异常要有运行日志，前端异常不能导致白屏。
- `GET /api/health` 必须能检查服务和数据库状态。
- 登录、档案修改、任务状态、告警处置、设备心跳、AI 复核和敏感数据访问必须留审计记录。
- 权限判断必须集中管理，前端隐藏菜单或按钮不能代替后端权限校验。
- 设备厂商字段必须先转换为标准事件，不能直接污染告警、任务和老人档案模块。
- AI 服务只能输出标准事件，是否生成告警由后端业务规则决定。

## 5. 数据库规范

- 当前试点优先使用 MariaDB / MySQL。
- 试点数据库可以部署在本地电脑或养老院局域网服务器，不要求第一阶段购买云数据库。
- 当前本机便携数据库运行目录为 `backend/.runtime/`。
- `backend/.runtime/`、数据库数据目录、日志、备份和临时文件不得提交 Git。
- 开发阶段可以使用 `DB_SYNC=true` 帮助快速建表，但生产或正式试点环境必须使用 `DB_SYNC=false`。
- 生产或正式试点环境不能依赖 TypeORM 自动同步表结构，必须使用迁移脚本。
- 每张核心业务表应有稳定业务编号，例如 `RES-001`、`TASK-001`、`ALERT-001`、`CAM-001`。
- 业务表应尽量保留创建时间、更新时间、状态和操作人等追溯字段。
- 后续必须补充 ERD、迁移脚本目录、迁移执行说明和回滚说明。
- 试点交付前必须制定备份策略，并实际做一次恢复测试，不能只确认“备份文件存在”。
- 云数据库只作为后续选项；切换前必须验证兼容性、网络安全、备份和迁移时间。

## 6. 前端规范

- `src/index.html` 作为管理端页面入口或应用外壳。
- 页面代码按业务拆分，页面只负责本页面渲染和事件绑定。
- API 地址、Authorization token、错误处理和后端不可用提示集中管理。
- 菜单、路由、页面标题、权限和页面模块必须有唯一映射。
- 每个菜单必须有唯一 `key` 和唯一 `path`。
- 当前路径只允许计算出一个 active menu key；不允许多个普通菜单同时高亮。
- 菜单 active 状态由 Router 统一计算，不允许每个菜单自己使用 `includes` 或宽泛 `startsWith` 判断。
- 不同菜单必须进入不同页面或不同业务视图，不能多个入口共用一个无意义占位内容。
- 页面未完成时，要显示该模块自己的建设中说明，不得伪装成其他页面。
- 404 和无权限页面不能错误高亮普通菜单。
- 刷新页面后，登录状态、当前路由和菜单高亮应保持一致。
- 空列表必须显示“暂无数据”或明确原因，不能留下空白区域。
- 正式业务数据必须来自后端和数据库。
- 后端不可用时，只允许使用明确标记的只读演示数据；写操作必须禁用或明确失败。
- 不得把演示数据、演示视频、固定指标或模拟结果说成真实后端数据或真实 AI 识别结果。
- 管理端优先适配电脑浏览器。
- 护理员端优先适配手机和平板网页。
- 当前阶段只做响应式网页，不开发原生 App。
- 前端文件过大时按职责拆分，但不得为了拆分引入新框架或破坏启动方式。

## 7. 后端规范

- 每个 NestJS 业务模块应包含 module、controller、service、dto 和 entity 等清晰边界。
- API 统一使用 `/api` 前缀。
- 登录使用 JWT；受保护接口必须校验 Bearer token。
- 角色、路由权限、数据范围和写操作权限必须由后端校验。
- 家属只能读取绑定老人数据；护理员只能读取职责范围内的数据。
- 所有新增、修改、状态流转和复核操作都要做写权限校验。
- 关键写操作必须写审计日志，至少记录操作人、角色、动作、资源、时间和摘要。
- 参数校验优先使用 DTO、`class-validator` 和全局 `ValidationPipe`。
- controller 不得堆复杂业务逻辑，也不得直接拼复杂数据库查询。
- HTTP 状态码必须正确使用：400 参数错误、401 未登录、403 无权限、404 不存在、409 冲突、422 业务校验失败、500 系统异常。
- 错误码使用英文大写下划线，例如 `VALIDATION_ERROR`、`AUTH_TOKEN_EXPIRED`、`ALERT_ALREADY_HANDLED`。
- 不得吞掉异常后返回成功。
- `GET /api/health` 必须能区分服务启动和数据库连接状态。
- 设备和 AI 上报接口后续必须增加设备密钥、签名或其他机器身份验证；当前没有这项能力时不得声称已达到生产安全要求。
- 正式角色名称和权限矩阵目前存在文档冲突，确认前不得擅自扩大任何角色权限。

## 8. AI 与摄像头边界

- 当前阶段以摄像头台账、配置预留、AI 事件记录和人工复核为主。
- 当前不训练模型，不执行真实 AI 推理，不承诺真实识别准确率。
- 当前不做复杂 RTSP 视频流生产级接入、转码、断流重连和多路视频墙。
- 浏览器不应直接播放裸 RTSP；未来需要视频网关转换为 HLS 或 WebRTC。
- 演示视频和模拟事件必须明确标注，不能说成实时监控或真实识别结果。
- AI 事件必须经过人工复核；AI、LLM 或模型建议只能辅助护理员、护士和医生判断，不能替代专业人员。
- 重大健康建议和告警处置不能由 AI 自动生效。
- 摄像头只考虑合法自有设备和公共安全区域，不连接陌生公共摄像头。
- 卧室、卫生间、洗浴区等私密区域不得随意安装或展示摄像头画面。
- 摄像头原始地址、账号和密码只允许最小权限人员访问，并必须记录审计日志。
- 家属不能查看公共摄像头、其他老人画面或未脱敏的视频数据。
- 视频、截图、测试数据集和 Demo Video 不得提交 Git。
- 真实视频网关、AI 推理、目标跟踪和模型训练必须另行立项、完成隐私评估并更新架构文档。

## 9. 隐私与合规规则

- 老人健康信息、护理等级、病史、用药、告警和康复记录按敏感数据管理。
- 家属姓名、电话、授权关系和反馈记录按个人信息管理。
- 护理员任务、操作记录、考核记录和响应轨迹只能按岗位授权查看。
- 摄像头视频、截图、RTSP 地址和设备账号属于高敏感数据。
- 位置轨迹、门禁记录和电子围栏数据只能用于明确授权的安全和照护目的。
- 告警记录和 AI 事件必须记录来源、时间、处置人、处置结果和人工复核结果。
- 家属只能看到绑定老人和机构明确授权的信息。
- 员工遵循最小权限原则，离职或岗位变化后要及时调整账号权限。
- 敏感数据访问、导出、修改和删除必须留下审计记录。
- 系统上线前必须明确数据收集目的、使用范围、授权说明、保存期限、访问人员和删除方式。
- 数据留存期限目前没有统一确认，按《待确认问题》处理；确认前不得无限期保留视频和轨迹。
- 日志不得记录明文密码、JWT token、完整身份证号、完整家属手机号、RTSP 密码或私密视频地址。
- 不得提交真实老人、家属、护理员隐私数据。
- 不得提交真实账号密码、API 密钥、访问令牌、数据库密码和私钥。
- 不得提交数据库运行文件、数据目录、真实备份文件和生产日志。

## 10. GitHub 提交与推送规则

GitHub 提交与推送规则以 AGENTS.md 中已存在的规则为准。每次任务完成后必须按 AGENTS.md 执行检查、提交和推送。如果 push 失败，必须明确说明原因，不得假装成功。

## 11. 试点交付规则

- 第一阶段优先选择 20–50 张床、小范围楼层或护理单元试点。
- 不要一开始就全院铺开，也不要同时接入全部硬件和外部系统。
- 优先跑通老人档案、护理任务、告警响应、设备台账、AI 事件人工复核、审计留痕和管理看板。
- 试点数据必须使用虚构或完成脱敏的数据。
- 交付前必须提供部署说明、环境变量说明、启动和停止说明、备份与恢复说明。
- 交付前必须提供院长、护士长、护理员和运维人员的培训说明。
- 交付前必须提供验收表、已知问题清单、应急人工流程和运维说明。
- 院方人员必须能按文档启动和停止系统。
- 护理员必须能完成任务查看、状态更新和告警响应。
- 管理层必须能查看来自数据库的看板或报表。
- 关键问题必须能通过审计日志和业务记录追溯。
- 系统、数据库或网络故障时必须保留人工照护、电话、对讲机等备用流程。
- 试点期间建议每日检查服务、数据库、设备状态和未处理告警，每周复盘问题。
- 试点通过后再决定是否扩展楼层、设备、家属服务和外部接口。

## 12. Codex 执行规则

- 每次开始任务前先读取根目录 `AGENTS.md` 和 `PROJECT_RULES.md`。
- 还要读取与当前任务直接相关的 README 和 `docs/` 文档，不要只凭聊天记忆修改项目。
- 开始前检查 Git 分支、工作区状态和已有未提交改动。
- 不确定时先说明风险、冲突和影响，不要擅自大改。
- 复杂任务先说明目标、范围、修改文件和验证方法。
- 修改前告诉用户准备改什么；修改后说明实际改了什么。
- 每次只做当前任务需要的改动，不顺手重构无关模块。
- 不要为了视觉炫酷、技术新潮或代码“看起来高级”破坏稳定性。
- 不得随意更换技术栈、增加重型依赖或改变目录边界。
- 不得删除、覆盖或回退用户和其他任务留下的未确认改动。
- 新功能先更新架构或项目文档，再搭最小骨架，再实现业务。
- 优先使用框架自带能力，不重复手写已有功能。
- 修改后运行相应构建、测试、健康检查或启动检查；无法运行时明确说明原因。
- 不要把未完成、未测试或仍被阻塞的内容说成已完成。
- 不要把演示数据、模拟账号、演示视频或固定指标说成真实数据。
- 不要破坏已有登录、权限、数据库连接、启动脚本和可运行页面。
- Git 提交和推送按 `AGENTS.md` 执行，失败时报告真实原因。

## 13. 阶段关卡规则

项目必须按“立项、规划、开发、测试、试点、上线、维护”逐步推进。前一阶段没有达到准出条件，不能把项目说成已经进入下一阶段。

### 13.1 立项完成标准

满足以下条件后，才算立项完成：

- 项目要解决什么问题已经写清楚。
- 第一代 MVP 要做什么已经写清楚。
- 当前服务对象和主要使用岗位已经写清楚。
- 当前明确不做的事项已经列出。
- 前端、后端、数据库和运行环境已经初步锁定。
- 隐私、安全、现场实施和技术风险已经初步识别。
- 仍有争议的项目名称、角色或指标已经标记为“待确认”。

### 13.2 规划完成标准

满足以下条件后，才允许正式开发：

- 架构文档已经存在，并与当前技术路线一致。
- 老人档案、护理任务、告警、设备、AI 事件、审计和看板的模块边界清楚。
- 数据库类型、部署位置、环境变量和生产迁移原则清楚。

- API 前缀、主要接口方向、返回格式和错误处理方向清楚。
- 试点机构、楼层或 20–50 张床的范围清楚。
- 本次开发计划说明了修改文件、验证方式和不做事项。

### 13.3 开发开始标准

Codex 满足以下条件后，才能开始写业务代码：

- 已读取项目根目录 `AGENTS.md`。
- 已读取项目根目录 `PROJECT_RULES.md`。
- 已读取本次任务直接相关的 README 和 `docs/` 文档。
- 已确认当前 Git 分支和远程跟踪关系。
- 已检查工作区是否存在用户或其他任务留下的未提交改动。
- 已向用户说明本次修改目标、范围、文件和验证计划。
- 已确认本次任务不需要擅自更换技术栈或引入重型依赖。
- 已确认不会覆盖、删除或误提交无关改动。

### 13.4 测试通过标准

同时满足以下条件，才能说本次开发通过测试：

- 后端服务能够启动，健康检查有明确结果。
- 前端页面能够打开，没有明显白屏。
- 本次涉及的核心接口能够访问，成功和失败场景都符合预期。
- 登录、退出、未登录拦截和角色权限正常。
- 正式业务页面读取后端和数据库数据，不用固定前端数据冒充。
- 关键写操作能够生成审计日志。
- 菜单只高亮当前页面，不出现多个普通菜单同时高亮。
- 不同菜单进入不同业务页面，不显示无关占位内容。
- 没有明显接口 500、死循环、重复提交或页面崩溃。
- 测试命令、结果和未覆盖风险已经记录。

### 13.5 试点开始标准

同时满足以下条件，才允许进入养老院试点：

- 试点账号和权限已经检查，默认密码不会用于正式生产。
- 系统不可用时的人工记录、电话、对讲机或纸质表格流程已经准备。

## 15. 发布与回滚规则

### 15.1 版本号规则

建议采用语义化版本：

```text
v0.1.0  MVP 骨架
v0.2.0  可试点版本
v0.3.0  试点优化版本
v1.0.0  正式交付版本
```

- 只修复兼容性 bug，一般修改最后一位，例如 `v0.2.0 -> v0.2.1`。
- 新增一个向后兼容的业务模块，一般修改中间一位，例如 `v0.2.1 -> v0.3.0`。
- 正式交付或出现重大不兼容变化，才评估修改第一位。
- 版本号只是建议，首个正式发布版本仍需项目负责人确认。

### 15.2 发布前检查

- [ ] Git 工作区干净，或所有保留的未提交文件已经明确说明。
- [ ] 当前分支和目标发布分支正确。
- [ ] 已更新本次变更记录。
- [ ] 核心功能和权限测试通过。
- [ ] 数据库已经备份。
- [ ] 数据库变化有迁移和回滚说明。
- [ ] 生产环境变量已经核对，但秘密没有写进文档或仓库。
- [ ] 端口、启动命令和健康检查地址已经确认。
- [ ] `.env`、日志、数据库文件、视频和真实隐私数据没有提交。
- [ ] 已知问题和上线风险已经告知相关人员。

### 15.3 发布记录

每次发布必须记录：

- 版本号。
- 发布日期。
- 修改内容。
- 影响模块。
- 数据库是否变化。
- 是否需要迁移，以及迁移命令。

### 15.4 回滚规则

- 上线失败时不能硬撑，满足回滚条件就及时回滚。
- 登录不可用、数据库异常、核心页面打不开、告警无法处理或任务状态无法流转时，应优先回滚。
- 回滚前先保护上线期间已经产生的新数据和日志。
- 回滚代码不能直接删除新产生的业务数据。
- 数据库回滚必须按迁移和备份方案执行，不能手工随意删表改表。
- 回滚后必须验证登录、健康检查、数据库、告警和任务功能。
- 必须记录失败原因、影响时间、处理过程、最终结果和后续预防措施。
- 不允许删除业务记录或审计日志来掩盖上线问题。

### 15.5 CHANGELOG 规则

- 根目录当前没有 `CHANGELOG.md`，建议在首次正式版本发布前创建。
- 每次正式版本必须更新 `CHANGELOG.md`，记录新增、修改、修复、数据库变化、已知问题和回滚说明。
- 本次规则整理不补造历史版本记录。
- 历史版本信息来源不清楚时标记“待确认”，不能凭印象编写。

## 16. 运维与故障处理规则

### 16.1 日常巡检

每天建议检查：

- [ ] 后端服务进程是否在线。
- [ ] MariaDB / MySQL 是否在线。
- [ ] `/api/health` 是否返回服务和数据库正常。
- [ ] 磁盘空间是否充足，日志和备份是否异常增长。
- [ ] 未处理告警是否堆积。
- [ ] 设备离线数量、低电量和长时间无心跳数量是否异常。
- [ ] 运行日志是否出现大量 error 或重复异常。
- [ ] 最近一次数据库备份是否按计划完成。

### 16.2 每周复盘

每周建议复盘：

- 护理任务完成率、超时原因和重复操作。
- 告警响应时间、未解决告警和升级情况。
- 告警和 AI 事件误报情况。
- 设备离线、故障和心跳异常情况。
- 护理员实际使用反馈。
- 院方管理层反馈。
- 经授权收集的家属反馈。
- 本周 bug、临时处理方法和下周改进项。

### 16.3 故障分级

- **P0：** 系统完全不可用、数据库损坏、核心数据丢失或存在严重隐私泄露风险。
- **P1：** 登录失败、告警无法处理、护理任务无法更新或核心权限失效。
- **P2：** 部分页面、部分接口或部分设备状态异常，但仍有人工替代方式。
- **P3：** 样式、文案、非核心交互或不影响业务流程的体验问题。

### 16.4 故障响应

- P0 必须立即优先处理，并同时启动人工应急流程和数据保护措施。
- P1 影响试点运行时必须尽快修复，不能等普通版本排期。
- P2 进入短期修复计划，并说明临时绕行方法。
- P3 可以进入后续优化清单，但仍要记录和跟踪。
- 所有故障必须记录发现时间、等级、原因、影响范围、处理过程、恢复时间和最终结果。
- 没有查清根因时要明确写“根因待确认”，不能编造结论。

### 16.5 人工应急流程

- 系统故障不能中断老人照护和真实告警响应。
- 系统不可用时，养老院必须切回人工记录、电话、对讲机、纸质任务单和纸质告警登记等备用流程。
- 护理员优先处理老人实际需要，不能为了修系统延误现场响应。
- 系统恢复后，由授权人员补录必要记录，并注明补录时间和原因。
- 补录不能覆盖原始纸质或人工记录，必要时保留两者对应关系。

### 16.6 运维边界

- 代码错误、接口错误、权限逻辑和部署脚本问题属于软件维护。
- 摄像头、手环、床垫、传感器和服务器硬件故障属于硬件维护。
- 断网、网络抖动、交换机、路由、防火墙和弱电问题属于网络维护。
- 账号使用错误、流程未执行、培训不足和现场分工不清属于院方操作或管理流程问题。
- 多方共同导致的问题要分别记录责任边界，不能把所有现场问题都归为软件 bug。

## 17. 数据留存与删除规则

### 17.1 数据分类

项目至少区分：

- 老人档案。
- 护理任务和护理记录。
- 告警记录和处置记录。
- 审计日志。
- 设备心跳和设备事件。
- AI 事件和人工复核结果。
- 摄像头配置、接入地址和权限信息。
- 视频、截图和事件证据。
- 家属反馈和授权关系。
- 员工操作记录。

### 17.2 留存原则

- 数据不是越多越好，只保存业务、试点、追溯和合规确实需要的数据。
- 敏感数据必须有明确目的、访问权限和留存期限。
- 视频、截图、位置轨迹和健康信息属于高敏感数据。
- 没有明确授权和留存规则前，不长期保存真实视频、截图和位置轨迹。
- 测试数据必须完全虚构或完成有效脱敏。
- 数据达到留存期限后，应按确认的归档或删除流程处理，不能无限期保留。
- 具体期限未确认前，不得自行编造保存天数或年限。

### 17.3 删除原则

- 删除敏感数据必须进行身份、角色和数据范围校验。
- 删除操作必须记录操作人、时间、原因、对象和结果。
- 普通用户不能随意删除老人档案、护理任务、告警、审计或其他关键业务记录。
- 删除前必须确认是否影响事件追溯、试点报告、备份、纠纷处理和合规要求。
- 有保留义务的数据优先使用归档或受控失效，不直接物理删除。
- 不允许删除日志、告警或护理记录来掩盖问题。

### 17.4 导出原则

- 数据导出必须限制角色、范围和用途。
- 导出老人、家属、护理员信息必须有明确业务目的和授权。
- 导出行为必须写审计日志，并记录导出人、范围、时间和用途。
- 导出文件不得随意通过个人微信、私人邮箱、公共网盘或未授权设备传播。
- 能满足用途时，优先导出脱敏数据而不是原始数据。
- 导出文件使用完成后，应按确认的期限和方式安全保存或删除。

### 17.5 待确认留存期限

以下期限尚未确定，统一放入末尾《待确认问题》：

- 老人档案保存多久。
- 护理记录保存多久。
- 告警记录保存多久。
- 审计日志保存多久。
- 视频和截图保存多久。
- AI 事件保存多久。
- 家属反馈保存多久。
- 员工操作记录保存多久。

## 18. 角色权限最终确认规则

### 18.1 当前状态

- 当前角色名称和权限矩阵仍存在文档冲突。
- 项目负责人确认前，不得擅自扩大任何角色的页面、数据或接口权限。
- 开发期间采用最小权限和默认拒绝原则。
- 前端菜单、页面和按钮权限只改善体验，不能代替后端接口权限。
- 角色规则仍未确认时，新写操作不得默认开放给多个角色。

### 18.2 需要最终确认的角色

至少需要确认以下岗位是否保留、合并或改名：

- 超级管理员 / 系统管理员。
- 院长 / 运营负责人。
- 护士长。
- 护理员。
- 医生 / 康复师。
- 设备管理员。
- 家属。
- 访客 / 只读用户。

### 18.3 每个角色需要确认的权限

每个角色必须逐项确认：

- 能查看哪些老人、楼层或护理单元。
- 能新增哪些数据。
- 能修改哪些字段和状态。
- 能确认、解决或标记哪些告警。
- 能查看和复核哪些 AI 事件。
- 能查看、配置或管理哪些设备和摄像头。
- 能导出哪些数据，导出范围多大。
- 能查看哪些审计日志。
- 能否创建、停用或管理账号。
- 能否查看健康信息、家属信息、视频地址和其他敏感信息。

### 18.4 权限开发原则

- 默认拒绝，只有明确授权后才允许。
- 写操作权限必须比读操作更严格。
- 家属权限必须最小化，只能访问明确绑定和授权的老人信息。
- 护理员和医生的数据范围必须由后端校验，不能只靠前端过滤。
- 设备和 AI 上报接口必须使用设备密钥、签名或其他机器身份验证；未实现时不能当成生产接口。
- 审计日志不能让普通用户随意查看、修改或删除。
- 生产环境不能展示默认账号和密码。
- 角色变更、账号停用和敏感权限授予必须留审计记录。
